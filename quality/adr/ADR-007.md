## ADR-007: Drop in-process network filtering — isolation-only sandbox
**Status:** Accepted (2026-04-28)
**Supersedes:** ADR-001 (host-side iptables proxy), ADR-006 (SNI anti-spoof)
**Driving scenario(s):** none directly — strategic simplification

### Context

claudius shipped a transparent proxy sidecar (`docker/proxy/`) that:
- Installed iptables/ip6tables PREROUTING REDIRECT rules on the Docker bridge for TCP
- Used NFQUEUE without `--queue-bypass` for UDP and ICMP (fail-closed)
- Read TLS SNI / HTTP Host headers and applied an fnmatch ACL (`CLAUDIUS_ALLOW`)
- Resolved SNI hostnames via DNS to detect SNI-spoofing attacks (ADR-006)
- Cleaned up its own chains on exit

This was ~600 lines of Python (`entrypoint.py`), a Dockerfile, a start script, a supervisor, and a chain-pruning helper. Maintaining it carried real cost:

- Python netfilterqueue bindings are platform-specific (Linux only)
- IPv4/IPv6 dual-stack added complexity (per-family chains, getsockopt for original dst)
- macOS support required `CLAUDIUS_NO_PROXY=1` — a feature flag that bypassed the entire central security claim
- SNI inspection couldn't authenticate hostnames against IPs cheaply (ADR-006 trade-off)
- Each new use case wanted a new entry in `CLAUDIUS_ALLOW` — high friction
- The wildcard `*:443/tcp` (often needed for real workflows) reduced filtering value to near zero

### Decision

The proxy sidecar is removed. claudius is now an **isolation sandbox**, not a **filtering sandbox**. Outbound network is unrestricted by design. Users who need network filtering apply it at the host (firewall, VPN, DNS sinkhole) or at the runtime layer (gVisor netstack policies if/when configured).

What stays:
- Per-session Docker network (still gives the docker-socket-proxy a place to live)
- Docker socket proxy (read-only by default, `CLAUDIUS_DOCKER_WRITE=1` opt-in)
- Capability drop, privilege drop, gVisor option, mount whitelist, managed CLAUDE.md, clipboard bridge

What goes:
- `docker/proxy/` directory
- `CLAUDIUS_ALLOW`, `CLAUDIUS_ALLOW_FILE`, `CLAUDIUS_ALLOW_PROFILE`, `CLAUDIUS_NO_PROXY` env vars
- `claudius logs` subcommand
- iptables-chain pruning in `claudius prune`
- Proxy image build paths (Makefile, `cmd_build`, `cmd_doctor`)
- `allow.example.ini`

### Consequences

+ ~330 lines of code deleted; `claudius.sh` shrinks from 439 → 336 lines; `docker/proxy/` (~600 lines Python) gone entirely
+ macOS support no longer requires a feature flag — claudius works the same on every Docker host
+ Setup time drops (no second image to build)
+ User mental model is simpler: "container + capabilities + gVisor", no allow-list to maintain
+ Eliminates a class of bugs (stale iptables chains, NFQUEUE crashes, SNI parser edge cases)
- The "Locked down by default" promise no longer covers network egress — Vision and Quality Requirements are revised accordingly
- Loss of differentiation vs. plain `docker run` with similar `--cap-drop` flags — the value proposition is now "curated sandbox + Docker-socket-proxy + clipboard bridge", not "filtered network"
- Users who relied on network filtering must adopt it at a different layer — README documents this

### Notes

The pentester-style CLAUDE.md instruction in the project root (asking the agent inside the container to attempt to escape) was the original motivation for filtering. With ADR-007, the test stance shifts: a successful escape from the proxy was never a meaningful guarantee anyway because `*:443/tcp` was almost always set. Isolation is the real boundary; the proxy was theatre that paid maintenance cost.
